# ADR-025f59f52: Fix Incorrect Assertions in edge_case_snapshot_tests

## Status
Proposed

## Context

The `edge_case_snapshot_tests.rs` file contains 4 tests with incorrect assertions that fail even when the renderer implementations are correct:

1. **`test_junit_empty_fields` (lines 361-362):** Asserts `<checkstyle>` root element but JUnit renderer uses `<testsuites>` (standard JUnit XML format)

2. **`test_csv_empty_fields` (line 433):** Asserts 4 consecutive commas `",,,,"` but actual output is `",0,,warn,,"` with `0` and `warn` breaking consecutive sequences

3. **`test_tsv_empty_fields` (line 501):** Asserts 5 consecutive tabs but actual output is `"\t0\t\twarn\t\t"` with values breaking consecutive sequences

4. **`test_sarif_unicode_characters` (line 244):** Asserts `&#x` OR `&amp;` encoding but the test message `"错误消息 with emoji 🎉"` contains no `&` character and UTF-8 characters don't trigger `&#x` encoding

All 4 tests are `#[ignore]`d pending issue #79, so these assertion bugs are masked.

## Decision

Fix the 4 incorrect assertions to match actual renderer output:

1. **JUnit:** Change `<checkstyle>` → `<testsuites>` (lines 361-362)
2. **CSV:** Change assertion to check for actual output pattern `",0,,warn,,"` (line 433)
3. **TSV:** Change assertion to check for actual output pattern `"\t0\t\twarn\t\t"` (line 501)
4. **SARIF:** Add `&` character to test message so `&amp;` encoding assertion can pass (line 230)

## Consequences

**Benefits:**
- Tests will correctly validate renderer behavior when un-ignored
- No implementation changes required
- Maintains correctness of existing implementations
- Removes masked bugs that could hide real issues

**Tradeoffs:**
- Tests become coupled to specific output patterns including numeric/string values
- SARIF test data requires modification

## Alternatives Considered

| Alternative | Rationale for Rejection |
|-------------|------------------------|
| Remove assertions entirely | Defeats purpose of tests - need to verify empty field rendering and encoding |
| Change implementations to match assertions | Implementations are correct per JUnit standard and CSV/TSV specs |
| Check for partial consecutive separators (2 commas/tabs) | Less semantically meaningful than checking specific output pattern |
