# ADR-55866964: Adopt `map_or` for `ScopeArg` to `Scope` Conversion

## Status
Accepted

## Context
GitHub issue #331 reports two instances of the `clippy::map_unwrap_or` lint in `crates/diffguard/src/main.rs` at lines 2079 and 2211 (actual lines: 2090 and 2222). Both occurrences are in functions that build `CheckReceipt` structs:

- `build_tool_error_receipt` (line 2076, lint at line 2090)
- `build_skip_receipt` (line 2208, lint at line 2222)

The pattern `args.scope.map(Into::into).unwrap_or(Scope::Added)` is flagged because `Option::map` followed by `Option::unwrap_or` requires two visits to the Option, whereas `Option::map_or` combines both operations into a single visit with inline default handling.

The `From<ScopeArg> for Scope` conversion is infallible (line 561), making this a pure optimization with no semantic change.

## Decision
Replace the `map + unwrap_or` pattern with `map_or` at exactly two locations in `crates/diffguard/src/main.rs`:

| Location | Before | After |
|----------|--------|-------|
| Line 2090 (in `build_tool_error_receipt`) | `args.scope.map(Into::into).unwrap_or(Scope::Added)` | `args.scope.map_or(Scope::Added, Into::into)` |
| Line 2222 (in `build_skip_receipt`) | `args.scope.map(Into::into).unwrap_or(Scope::Added)` | `args.scope.map_or(Scope::Added, Into::into)` |

The third `map_unwrap_or` instance at line 2061 (`cockpit_error_detail`) uses `unwrap_or_else`, not `unwrap_or`, and is unrelated to scope resolution. It is explicitly out of scope for this ADR.

## Consequences

### Benefits
- **Performance**: `map_or` compiles to a single branch vs. two for `map` + `unwrap_or`
- **Clippy compliance**: Satisfies `clippy::map_unwrap_or` lint
- **Codebase precedent**: `clippy_refactor_test.rs` documents a prior `map_unwrap_or` fix, establishing this as an accepted pattern

### Tradeoffs / Risks
- **None significant**: Purely mechanical refactor with zero behavioral change
- **Scope creep risk**: The third lint at line 2061 must not be fixed as part of this work item

## Alternatives Considered

1. **Leave as-is**: Unacceptable — lint is actively flagged and codebase precedent shows this pattern has been fixed before.

2. **Use `unwrap_or_else`**: Does not satisfy `map_unwrap_or` lint. The `unwrap_or_else` variant is semantically different (lazy default) and is at line 2061, which is out of scope.

3. **Extract helper function**: Unnecessary complexity. Does not improve performance or readability beyond `map_or`.

## Verification
- `cargo clippy -p diffguard -- -W clippy::map_unwrap_or` must report zero warnings
- `cargo test -p diffguard` must pass
- Exactly 2 lines modified in `crates/diffguard/src/main.rs`
